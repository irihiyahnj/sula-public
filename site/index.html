How It Works

The actual launch flow should feel like this.

01

Read The URL

Open the launch page and `sula.json`, then resolve the canonical launcher source instead of guessing a package or global binary.

read /launch/
read /sula.json
resolve canonical launcher
02

Classify The Project

Decide whether the project is a new onboarding target or an existing Sula consumer.

if .sula/project.toml exists
  review consumer
else
  run onboard
03

Route Safely

New projects go through guided onboarding. Existing consumers go through `doctor --strict` and `sync --dry-run` review.

onboard
or
doctor --strict
sync --dry-run
04

Explain The Promise

Tell the user what Sula will manage, where files will go, what source was used, and what to run next.

report source used
report managed scope
report file routing
report next commands

Canonical Launcher

The site should be able to launch Sula without local guessing.

  • The canonical machine descriptor lives at /sula.json.
  • The canonical human-readable launch contract lives at /launch/.
  • The canonical bootstrap shim lives at /launch/bootstrap.py.
  • When local tooling is missing, agents should resolve source from the site contract instead of searching npm.

Immediate Manual Fallback

The launcher can still be called directly.

python3 bootstrap.py --project-root /path/to/project
python3 bootstrap.py --project-root /path/to/project --accept-suggested --approve

If the project is already adopted, the launcher should review it instead of trying to adopt again.

Managed Boundary

Sula should manage the operating system, not absorb project truth.

  • Managed files can be synced later from Sula Core.
  • Scaffold files are starters and stay owned by the project after first generation.
  • Status, change history, release notes, and incident notes remain durable project memory.
  • Approval is always required before adoption writes into a repository.

Current State

The site is in development, but the bootstrap contract is already usable.

  • The public domain can host this exact static page set.
  • The canonical bootstrap flow already exists in the Sula CLI.
  • The source repository is not ready for direct public release from current history yet.
  • The future public source should come from a clean Sula-only history.

What You Get

Normal usage already delivers concrete operational benefits.

Faster session startup

New agents stop behaving like empty-context contractors and start from explicit project rules.

Durable project memory

Rules, current status, decisions, releases, and incidents have fixed locations instead of chat-only history.

Safer shared upgrades

`sync --dry-run`, version locks, canaries, and strict doctor checks reduce blind rollout risk.
